제품을 만드는 운영체제

dev-os는 랩탑 한 대 안에서 1명의 계획가 AI + 4명의 전문 sub-agent + 1명의 외부 구현가가 역할을 나눠 일하게 만드는 개발 운영 체계(Host-OS)입니다. 당신이 "뭘 만들지"만 말하면, 시스템이 티켓으로 쪼개고 → 구현하고 → 적대적으로 검토하고 → 통과한 것만 합칩니다.

한 문장으로

dev-os가 정확히 뭔가요?

dev-os는 "AI에게 코드를 막 시키는 것"을 "공장 라인처럼 돌리는 것"으로 바꾸는 규칙 + 도구 세트입니다. 핵심은 한 사람의 AI가 모든 걸 하지 않는다는 것 — 계획하는 AI, 만드는 AI, 트집 잡는 AI가 따로 있고, 각자 권한이 물리적으로 분리돼 있습니다.

🏠 Host-OS

엔진·에이전트·규칙(doctrine)은 랩탑의 ~/dev-os 한 곳에만 존재합니다. 모든 제품 프로젝트가 이 한 벌을 공유 — OS를 고치면 모든 프로젝트에 즉시 반영(sync 불필요).

🎫 티켓 중심

모든 구현 작업은 QUEUE.yaml 안의 티켓이 됩니다. 1 티켓 = 1 PR. 티켓은 "무엇을·왜"를 담고, 어떻게 만들지는 구현 AI가 정합니다.

🎭 오케스트레이션

메인 AI(CLAUDE1)는 직접 코드를 안 칩니다. 대신 sub-agent를 불러 일을 시키고, 결과를 별도 검토 AI에게 넘기는 지휘자 역할만 합니다.

바이브코더에게 핵심

당신은 "이거 만들어줘"라고 말하는 제품 책임자(PM) 자리에 앉습니다. 코드를 직접 안 봐도, 시스템이 "happy path만 말하는 비개발자의 빈틈"을 자동으로 메워줍니다(에러 케이스 강제, 보안 감사 자동, 적대적 검토). 그래서 "대충 시켜도 꼼꼼하게 나오는" 구조입니다.

동기

왜 이런 게 필요한가

AI에게 그냥 "앱 만들어줘"라고 하면 빠르게 뭔가 나오지만, 흔히 이런 일이 생깁니다 👇

🔥 AI에게 막 시킬 때

  • 잘 되는 경로(happy path)만 짜고 에러 처리는 빠짐
  • "되는 것 같아요"라고 말하고 실제론 안 됨
  • 자기가 짠 코드를 자기가 검토 → 객관성 0
  • 인증·결제 같은 위험 코드도 그냥 통과
  • 범위가 슬금슬금 늘거나, 반대로 "v1은 일단 대충"으로 줄어듦
  • 무슨 결정을 왜 했는지 기록이 안 남음

✅ dev-os가 강제하는 것

  • 성공 케이스 DOD가 있으면 에러 케이스 DOD 필수
  • 실행 증거 없이는 "완료" 선언 금지 (Iron Law 3)
  • 구현 AI ≠ 검토 AI. 검토자는 고칠 권한 자체가 없음
  • auth/payment/권한 티켓은 보안 감사 자동 발동
  • "임시", "나중에", "v1으로 일단" 같은 축소 어휘 금지
  • 모든 디스패치가 로그·상태 전이로 기록됨
철학: Boil the Lake

"AI는 완성도의 한계 비용을 거의 0으로 만든다. 완전한 구현이 단축본보다 몇 분 더 걸릴 뿐이라면, 매번 완전한 것을 한다." — 즉 dev-os는 "대충 빨리"가 아니라 "이왕이면 끝까지"를 기본값으로 둡니다. 테스트는 미루지 않습니다(가장 싸게 끝낼 수 있는 호수).

멘탈 모델

딱 3가지 개념만 잡으면 됩니다

1Host-OS — 엔진은 하나, 프로젝트는 여럿

예전에는 프로젝트마다 OS 설정을 복사해서 동기화했습니다. 지금은 호스트(~/dev-os) 한 곳에만 엔진·에이전트·규칙이 있고, 각 제품은 ~/dev-os/projects/ 아래의 독립 git 레포로 들어옵니다. 프로젝트는 자기 제품 코드와 자기 작업 상태만 들고, 나머지는 호스트가 빌려줍니다.

  • 엔진은 1곳: OS를 고치면 전 프로젝트에 즉시 적용 — 동기화 명령 없음
  • 프로젝트는 독립: 각자 git 레포 + 각자 task 상태, 따로 배포
  • 진입은 하나로: os3 open <프로젝트> 한 줄이면 호스트 설정을 달고 들어감

2티켓 — 모든 일의 최소 단위

아이디어/요구사항(PRD)은 곧장 코드가 되지 않습니다. 먼저 티켓으로 쪼개집니다. 티켓은 goal(무엇), context(왜·조사 결과), dod(완료 기준), files(건드릴 파일), verify(검증법) 같은 필드를 가진 자기완결적 작업 명세입니다. 1 티켓 = 1 PR.

3오케스트레이션 — 지휘자와 연주자의 분리

메인 AI(CLAUDE1)는 악보를 쓰고 지휘만 합니다. 실제 연주(구현)는 sub-agent가, 평가(검토)는 또 다른 sub-agent가 합니다. 이 분리가 객관성과 추적 가능성을 만듭니다.

철의 규칙 (Rule 1)

메인 AI는 production 코드를 직접 작성하지 않습니다. "내가 빨리 해버리면 더 간단한데?"라는 생각이 들면 — 멈추고 티켓을 만들어 구현 AI에게 넘깁니다. 이건 협상 대상이 아니라 시스템의 토대입니다.

큰 그림

아키텍처 한눈에 보기

두 개의 층이 있습니다 — (A) 파일 시스템 층(호스트 + 프로젝트들)과 (B) 세션 안의 에이전트 층(지휘자 + 연주자 + 평가자).

등장인물

누가 무슨 일을 하나

각 에이전트는 다른 모델을 쓰고, 다른 권한(tools)을 가집니다. 권한 분리는 "신뢰"가 아니라 "구조"로 강제됩니다 — 예를 들어 검토자는 Edit/Write 툴 자체가 없어서 고치고 싶어도 못 고칩니다.

CLAUDE1 (main)Opus계획·지휘사용자가 직접 대화하는 상대

PRD를 받아 티켓으로 분해하고, 조사(라이브러리 최신 API 등)를 하고, 어떤 에이전트에게 일을 줄지 정합니다. 직접 코드를 쓰거나 결과를 직접 OK 하지 않습니다. 고칠 수 있는 건 규칙·문서(devos/, .claude/)뿐.

buildersonnetRead·Edit·Write·Bash세션 안 구현가

티켓을 받아 실제 코드를 짭니다. 모호하거나 감성이 중요한 UI · 새 UX 흐름 · 탐색적 프로토타입에 우선 투입. 티켓의 files: 범위 밖은 절대 안 건드립니다.

revieweropusREAD-ONLY적대적 PR 검토자

구현이 끝나면 트집을 잡으러 옵니다. 단언(assertion)이 구체적인지, DOD↔테스트가 1:1로 매핑되는지, 에러 케이스가 실제로 테스트됐는지, 범위를 벗어났는지 검사. 발견만 하고 보고만 합니다(고칠 권한 없음). 확신이 70% 미만이면 uncertainty=true로 b'를 부릅니다.

designersonnetREAD-ONLYUI/UX 1차 필터

UI 티켓에서 일관성·정보 위계·상태 누락(로딩/빈/에러)·여정 갭·접근성·비개발자 보호를 점검. apps/web/를 건드리는 티켓에 자동 호출.

securityopusREAD-ONLYOWASP/STRIDE 감사관

인증·결제·권한·외부 입력을 다루는 티켓(security_audit:true, 키워드 감지 시 자동 강제)에 발동. OWASP A01~A10 + STRIDE 체크.

CODEX외부 CLIsubprocess·sandbox플랫폼 구현 + 교차검증

별도 프로세스로 도는 외부 벤더 구현가. 코드 비중이 큰 백엔드/API/공유 패키지/인프라/테스트/마이그레이션과, 검토자가 불안해할 때의 교차모델 2차 의견(b')을 담당. 다른 회사 모델이라 "벤더 다양성" 안전망이 됩니다.

builder vs CODEX, 누구한테 줄까?

builder(세션 내) → 답이 열려있는 제품 UI, 새 UX, 느낌이 중요한 작업.   CODEX(외부) → 백엔드/데이터/공유패키지처럼 코드 비중 큰 작업, 테스트·인프라·마이그레이션, 정책 강제, 객관적 버그 수정. 이 라우팅은 티켓의 owner/files로 자동 결정됩니다.

생애주기

"만들어줘"가 코드가 되기까지

당신이 던진 아이디어가 실제로 머지되기까지의 전체 흐름입니다.

1

진입 — os3 open meation

한 줄이면 프로젝트 폴더로 이동 + 호스트 설정(CLAUDE.md·에이전트·권한)을 달고 Claude가 켜집니다.

2

PRD 제출 → 티켓 분해

당신이 "이런 기능 원해요"를 말하면, CLAUDE1이 PRD intake 체크리스트로 빈틈(에러·엣지·보안)을 캐묻고 티켓으로 쪼갭니다. UI라면 designer가 사전 검토.

3

플랜 승인 — os3 approve

분해된 계획은 devos/plans/pending/에 저장되고 당신의 승인을 기다립니다. 마음에 안 들면 os3 reject "이유"로 되돌립니다. 승인 전에는 QUEUE에 안 들어갑니다.

4

디스패치 — os3 dispatch T-XXX

티켓의 owner에 따라 builder(세션 내) 또는 CODEX(subprocess)로 라우팅. 구현가가 코드를 짜고 Done / Block / Log를 돌려줍니다.

5

게이트 + 적대적 검토

pr-check 게이트(시크릿 스캔·범위·계약 동기화·로그) 통과 후, 티켓 유형별로 reviewer / security / designer가 읽기 전용으로 검토. 불안하면 CODEX 교차검증(b')이 자동 발동.

6

판정 → 상태 갱신

모든 의견을 모아 BLOCKER 우선으로 최종 판정. BLOCKER면 blocked + 질문 큐 기록, 아니면 done + 세션 로그 작성.

7

정리 — os3 archive

done 티켓은 ARCHIVE.yaml로 이관(토큰 누적 방지). 호스트는 git push로 랩탑 간 동기화.

엔진 내부

디스패치 7단계 — 검토가 실제로 도는 법

위 5~6단계가 내부적으로 어떻게 굴러가는지, 조금 더 자세히. 이건 /dispatch T=X를 받았을 때 CLAUDE1이 따르는 결정적(deterministic) 절차입니다.

단계하는 일실패하면
1. Lookup + pre-flightos3 lookup {id}로 티켓 조회(QUEUE→ARCHIVE). status==todo인지, deps가 다 done인지, files 경로가 실제 있는지 검사blocked + 사유 출력
2. Owner 라우팅owner가 BUILDER → 세션 내, CODEX → subprocess, CLAUDE1 → 직접(문서/정책 티켓 한정)unknown owner → 중단
3. 구현 디스패치builder는 Agent()로, CODEX는 dispatch-codex로. 부트 규칙(AI-core.md)을 prompt에 인라인 주입해 라운드트립 절약timeout/Block → blocked + Q-id
4. pr-check 게이트시크릿 스캔(gitleaks) · 계약 동기화 · 범위 가드 · 세션 로그 존재 · TDD 첫 커밋 게이트게이트 실패 → blocked
5. 검토 체인 (5분기)티켓을 critical / ui / backend / docs로 분류해 필요한 검토자만 호출. 게다가 5% 무작위 표본은 평소 가벼운 경로도 풀 검토로 끌어올림(안전망)검토자 timeout → WARNING 기록 후 진행
6. Fan-in + b' 트리거의견 취합. 아래 정량 조건 중 하나라도 맞으면 CODEX 교차검증 자동 발동CODEX timeout → reviewer 단독 + fallback 경고
7. 최종 처리 + 결과 헤더BLOCKER 우선 판정 → 상태 갱신 + 로그. 표준 결과 헤더로 사용자에게 보고QUEUE 쓰기 실패 → 수동 복구 Q-id

b'(교차모델) 자동 발동 조건 — 평소엔 비용 0, 의심될 때만

cross_model == true          # 티켓이 명시 요청
security_audit == true       # auth/payment/권한/외부입력
ethos == 'high'              # 삭제·결제·인증 등 위험 키워드 감지
reviewer.confidence < 0.7    # 검토자 스스로 불확실
blocker_count >= 1           # BLOCKER 발견 → 2차 검증
warning_count >= 3           # WARNING 누적
security.findings 1건 이상    # 보안 지적 있음
→ 하나라도 참이면  os3 cross-model-codex {id}  (60초 타임아웃)
표준 결과 헤더

자료구조

티켓 해부도

티켓은 devos/tasks/QUEUE.yaml 안의 YAML 블록입니다. 모든 작업이 이 모양을 따릅니다. 자기완결적이어야 합니다 — 구현 AI가 티켓 + SSOT만 읽고 실행할 수 있어야 합니다.

- id: T-AUTH-LOGIN-01          # 고유 ID
  status: todo                 # 새 티켓은 반드시 todo (그래야 디스패처가 집음)
  owner: CODEX                 # BUILDER | CODEX | CLAUDE1(문서 한정)
  impl_owner: CODEX            # 구현 담당
  test_owner: CODEX            # 테스트 담당 (cross-test 시 분리)
  tdd: required                # required | skip | self-evident
  security_audit: true         # auth/결제/권한이면 자동 true
  cross_model: false           # 임계경로면 true → CODEX 2차 의견
  goal: |                      # 무엇을 (행위 요구사항) — CLAUDE1이 작성
    POST /auth/login 구현
  context: |                   # 왜 + 조사 결과 (라이브러리 최신 API 등)
    bcrypt 12 rounds, JWT 15분 만료...
  constraints: |               # 기술 제약
  dod:                         # 완료 기준 — 입력+기대출력으로 검증가능하게
    - 'valid 자격증명 → 200 + access/refresh JWT'      # 성공
    - 'wrong password → 401 + "invalid credentials"'  # ★에러 필수★
    - 'missing email → 400 + validation error'        # ★에러 필수★
  files:                       # 건드릴 파일 = 소유권 범위 (밖은 금지)
    - apps/api/src/auth/login.py
    - tests/test_auth_login.py
  verify: |                    # 완료 확인법
    python3 -m pytest tests/test_auth_login.py -v
  deps: []                     # 선행 티켓
  gates: [pr-check]            # 완료 후 검증 단계

DOD 작성 규칙 — 가장 중요

❌ 나쁜 DOD

  • "인증이 잘 작동함"
  • "에러 처리가 적절함"
  • assert response (참 여부만 봄)

✅ 좋은 DOD

  • "valid 자격증명으로 로그인 → 200 + JWT"
  • "만료 토큰 → 401 + 에러 메시지"
  • assert response.status_code == 401
철칙

성공 케이스 DOD가 하나 있으면, 실패/에러 케이스 DOD가 반드시 함께 있어야 합니다. 이게 "비개발자가 happy path만 적는 빈틈"을 시스템 차원에서 메우는 핵심 장치입니다.

상태(status) 값

값의미
todo새 티켓. 디스패처는 이것만 집습니다.
doing작업 중
code_ready구현 끝, 게이트/리뷰/PM 결정이 남음
needs_pm제품 판단·취향 결정·최종 수용이 필요 (당신 차례)
done게이트·리뷰·결정·기록 모두 닫힘
blocked막힘 (질문 큐에 사유)
parked보류

ready·pending·queued 같은 값은 조용히 무시됩니다. 정해진 값만 쓰세요.

신뢰의 구조

안전장치 — "AI를 믿어서"가 아니라 "구조로 막아서"

Iron Laws — 양보 불가 5계명 (모든 에이전트 공통)

  1. 실패하는 테스트 없이 production 코드 없다 (비즈니스 로직 한정)
  2. 근본 원인 조사 없이 수정 없다 — 증상 패치 금지. 재현 → 원인 → 수정 → 재현 실패 검증
  3. 새 검증 증거 없이 "완료" 선언 없다 — "should work", "probably fine" 금지
  4. 적대적 검토 통과 없이 머지 없다 — 인증/결제/권한 티켓은 BLOCKER 0건이어야
  5. 티켓에 범위 축소 어휘 없다 — "v1으로 일단", "임시", "나중에", "TODO" 등 금지어 검사

🔒 권한으로 강제된 객관성

reviewer·designer·security는 읽기 전용 툴만 가집니다. 코드를 고치고 싶어도 물리적으로 불가능 — 그래서 "자기가 짠 걸 자기가 검토하는" 객관성 붕괴가 원천 차단됩니다.

🤝 b' 교차모델 안전망

한 회사 모델은 공통 맹점이 있을 수 있습니다. 검토자가 불안하거나 위험 티켓이면 다른 벤더(CODEX)가 2차 의견을 냅니다. 평소엔 비용 0, 의심될 때만.

🚪 pr-check 게이트

스택과 무관하게 모든 티켓에: 시크릿 스캔(gitleaks) · 계약 문서↔코드 동기화 · 티켓 범위 밖 파일 가드 · 세션 로그 존재 · TDD 첫 커밋 검사.

🧑‍🦽 비개발자 보호

PRD intake에서 도메인별 누락 항목을 강제 질문하고, 위험 영역엔 보안 감사를 자동 강제합니다. "사용자가 happy path만 적는" 패턴을 시스템이 보완합니다.

ETHOS — 판단이 충돌할 때의 최종 결정권

규칙이 서로 부딪히면 devos/ETHOS.md가 결정합니다. 우선순위는 명확합니다:

  1. 사용자의 명시적 지시 (대화/CLAUDE.md) — 최우선
  2. ETHOS.md (Iron Laws + 비개발자 보호)
  3. devos/AI.md 운영 규칙
  4. Superpowers 스킬 + 플러그인 기본값
  5. 기본 시스템 프롬프트 — 최하위
실제 사고에서 배운 것 (Rule 9)

명령어 치트시트 (os3)

모든 것은 단일 진입점 bin/os3를 거칩니다. PATH에 등록하면 어디서든 os3 ...로 씁니다. osn은 호환 alias.

호스트 레벨 (프로젝트 횡단)

os3 overview                  # 전 프로젝트 todo/doing/blocked + 피드백 집계
os3 projects                  # 등록된 프로젝트 목록
os3 register <name> <path>     # 프로젝트를 호스트 레지스트리에 등록
os3 open <name>               # 프로젝트 세션 진입 (cd + 설정 주입 + claude)
os3 feedback "<text>"         # OS 중앙 피드백 백로그에 기록

프로젝트 레벨 (--project <name> 또는 프로젝트 폴더 안)

os3 status        # 상태 + archived 카운트      os3 queue         # 활성 티켓 목록
os3 next          # 다음 디스패치 가능 티켓     os3 pending       # 승인 대기 플랜
os3 approve       # 플랜 승인                 os3 reject "이유"  # 플랜 반려
os3 lookup T-XXX  # 티켓 전체 YAML 출력        os3 logs          # 세션 로그 요약

os3 dispatch T-XXX        # 단일 티켓 (owner 자동 라우팅)
os3 dispatch-codex T-XXX  # CODEX 전용 subprocess
os3 dispatch-all          # 전체 todo 티켓
os3 dispatch-next         # 자동 선택 + 디스패치

os3 verify T-XXX          # 티켓 verify 실행
os3 pr-check              # 베이스라인 게이트 파이프라인
os3 cross-model-codex T-XXX  # b' 교차모델 2차 의견
os3 set-status T-XXX ...  # 사유와 함께 상태 갱신
os3 close T-XXX           # verdict 기록 + done으로 전진
os3 archive               # done 티켓 → ARCHIVE.yaml
os3 dashboard             # 로컬 읽기 전용 티켓 대시보드 (127.0.0.1:8787)
os3 cost-report           # 채널별 사용량 + 비용 추정
대시보드

os3 dashboard는 stdlib만으로 도는 로컬 읽기 전용 칸반 보드를 http://127.0.0.1:8787에 띄웁니다. 빌드 단계 없는 정적 프론트엔드 — 외부 의존성 0. 전 프로젝트의 티켓 상태를 브라우저로 한눈에 봅니다.

실전

새 프로젝트 추가하기

프로젝트는 ~/dev-os/projects/ 아래의 독립 git 레포여야 합니다. OS는 이미 호스트에 있으니 들고 올 게 없습니다 — 빈 레포 + 마커 + 등록 한 줄이면 끝.

복사 주의

아래는 줄 단위로 복사하세요. 일부 zsh가 # 주석을 명령으로 해석해 에러가 납니다. 한 번에 붙이려면 먼저 setopt interactive_comments.

cd ~/dev-os/projects
git init newtool && cd newtool
printf 'name: newtool\n' > .os3.yaml      # 프로젝트 마커 (필수 — cwd 인식용)
mkdir -p devos/tasks                      # 자기 task 상태 위치
cd ~/dev-os
os3 register newtool projects/newtool     # 호스트 레지스트리 등록
os3 open newtool                          # 작업 시작

기존 레포면 git init 대신 git clone <url> ~/dev-os/projects/newtool. 반드시 projects/ 아래에 둬야 호스트 .claude 자동 발견이 됩니다.

최초 1회 설정

git clone <your-os-repo> ~/dev-os && cd ~/dev-os    # 호스트 OS 클론
pip install -r requirements.txt                     # 파이썬 의존성
echo 'export PATH="$HOME/dev-os/bin:$PATH"' >> ~/.zshrc && source ~/.zshrc
os3 projects                                        # 동작 확인

데일리 루틴

cd ~/dev-os && git pull && os3 overview   # 출근: 전체 현황
os3 open meation                          # 작업: 프로젝트 진입 → PRD → 승인 → 디스패치
os3 archive --project meation             # 퇴근: done 정리
cd ~/dev-os && git push                   # 호스트 동기화 (각 프로젝트는 자기 remote로 따로)

바이브코더의 질문

FAQ

Q. 코드를 하나도 몰라도 쓸 수 있나요?

"무엇을 원하는지"를 말할 수 있으면 됩니다. CLAUDE1이 빈틈을 캐묻고, 시스템이 에러 케이스·보안·테스트를 강제합니다. 다만 제품 판단(이게 맞는 방향인가)과 최종 수용(이게 내가 원한 거 맞나)은 당신 몫입니다 — 그게 PM 자리입니다.

Q. 왜 메인 AI가 직접 안 짜요? 더 빠를 텐데.

"빨리"의 대가가 "객관성 붕괴 + 추적 불가"입니다. 구현가와 검토자를 분리해야 검토가 의미 있고, 티켓으로 남겨야 무슨 결정을 왜 했는지 추적됩니다. "내가 해버리면 간단" 충동은 시스템이 명시적으로 STOP 신호로 규정합니다.

Q. OS를 고치면 프로젝트마다 반영해야 하나요?

아니요. 엔진이 호스트 한 곳에만 있어, 고치면 다음에 어느 프로젝트를 열든 자동 적용. sync 명령이 없습니다.

Q. --project를 매번 써야 하나요?

프로젝트 폴더 안에서는 .os3.yaml 마커로 자동 인식되어 생략 가능. 호스트 루트 등 밖에서만 --project <name> 명시.

Q. 티켓이 막혔어요(blocked).
