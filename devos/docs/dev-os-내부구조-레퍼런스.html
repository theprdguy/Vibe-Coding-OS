§0한 장 요약

dev-os는 "AI에게 코드를 시키는 일"을 감사 가능한 상태 기계(auditable state machine)로 바꾼 것입니다. 모든 진실은 devos/ 아래 파일에 있고, 일은 티켓으로만 움직이며, 누가 무엇을 왜 했는지는 상태 전이 기록 + 세션 로그 + 질문 큐에 남습니다.

시스템을 한 문단으로:

사용자(PM)가 의도를 말한다 → CLAUDE1이 그것을 티켓으로 분해하고 빈틈(에러·보안·여정)을 캐묻는다 → 사용자가 plan을 승인한다 → 티켓의 owner에 따라 builder(세션 내) 또는 CODEX(외부 프로세스)가 구현한다 → pr-check 게이트 + 읽기 전용 reviewer/security/designer가 적대적으로 검토한다 → 위험·불확실하면 다른 벤더(CODEX b')가 2차 검증한다 → BLOCKER가 없으면 done, 있으면 blocked + 질문 큐(Q-XXX)로 사용자에게 결정을 넘긴다 → done 티켓은 ARCHIVE.yaml로 이관되고 세션은 8축 자가질문 회고로 닫힌다.

이 문서의 사용처

당신이 "바이브하게" 지은 구조를 역설계해 적은 것입니다. 각 절 끝의 놓치기 쉬움 박스가 직접 안 짠 자동 동작을 짚어줍니다.

§13층 정신모델

시스템은 세 겹입니다. 위 두 겹은 "무엇을 한다"이고, 맨 아래가 "그걸 어기지 못하게 한다"입니다. 대부분의 안전성은 3층에서 옵니다.

층구성역할저장/구현 위치
1. 진실(State)파일 기반 SSOT프로젝트가 무엇이고 지금 어디까지 왔는지를 기억devos/** (YAML/MD)
2. 행위자(Agents)CLAUDE1 + sub-agent 4 + CODEX계획·구현·검토를 수행.claude/agents/*, AGENTS.md
3. 강제(Enforce)권한·훅·게이트·금지어·잠긴결정규칙을 "신뢰"가 아니라 "구조"로 막음hooks, tools allowlist, pr-check, grep

핵심 원리 하나: 역할과 권한이 분리되어 있고, 권한은 도구(tool) 차원에서 물리적으로 제한된다. 검토자는 Edit/Write 툴 자체가 없어서 "고치고 싶어도 못 고친다" — 그래서 자기검토에 의한 객관성 붕괴가 원천 차단됩니다.

두 개의 교리 레이어 — docs/ vs devos/

"규칙"은 한 곳이 아니라 두 층에 삽니다. 이걸 구분하면 헷갈리지 않습니다:

레이어성격예
docs/ · docs/policy/정식 교리/정책 reference (대부분 status: draft). "OS3는 이래야 한다"는 규범. 모드·역할권한·웨이버·티켓스키마의 설계 원천OPERATING_DOCTRINE · MODE_GATE_MATRIX · ROLE_AUTHORITY_MATRIX · MODEL_ROUTING_POLICY · WAIVER_FORMAT · TICKET_SCHEMA
devos/ · .claude/라이브 운영 SSOT + 에이전트가 실제로 읽는 룰. 헌법(AI.md)·운영룰(CLAUDE.md)·상태·티켓·로그AI.md · CLAUDE.md · QUEUE.yaml · CONTEXT.md · logs/
놓치기 쉬움

docs/policy/는 "이렇게 하자"는 설계 문서이고, 그중 일부만(예: 모드 게이트) 이미 dispatcher.py에 구현되어 라이브로 강제됩니다. 나머지(웨이버 registry, 일부 역할)는 문서엔 있고 코드엔 부분 적용입니다. "규칙이 안 먹는다" 싶으면 그게 docs/의 draft 교리인지, devos/+코드로 실제 집행되는지부터 확인하세요. 이 레퍼런스 §10–11이 그 다리를 놓습니다.

§2SSOT 위계 — 진실이 충돌할 때

여러 파일이 서로 다른 말을 하면, 정해진 우선순위로 믿습니다. 위가 강합니다. (출처: AI.md § SSOT Precedence)

1. PROJECT_STATE.md ← 마일스톤·진행률의 최종 진실 2. docs/API_CONTRACT.md docs/UI_CONTRACT.md ← 인터페이스 계약 (코드보다 우선) 3. docs/ADR/* ← 아키텍처 결정 기록 (long-form) 4. tasks/QUEUE.yaml ← 활성 티켓 상태 5. 코드 ← 실제 구현 6. 세션 로그 (devos/logs/) ← 무슨 일이 있었나 7. 채팅 로그 ← 가장 안 믿음 (재현·검증 불가)
왜 코드가 4위가 아니라 5위인가

"계약이 코드보다 위"라는 뜻은: API/UI 동작을 바꾸려면 계약 문서를 먼저 고치고 코드를 맞춥니다(contract-first). 코드가 계약과 어긋나면 코드가 틀린 것으로 간주하고, reviewer가 "contract sync" 항목에서 BLOCKER를 냅니다.

Locked Decisions (D-XX) — 절대 어기면 안 되는 결정

CONTEXT.md 안에 사는 특수 레코드. 일반 결정과 다른 점: 위반이 grep/파일검사로 객관 검출되고, 위반 티켓은 waiver 없이 reviewer가 자동 BLOCKER로 막습니다. 등재 자격이 까다로워서(4조건) 인위로 만들지 않습니다 — 현재 단 1건:

- id: D-01
  status: locked
  decided: 2026-05-16
  summary: HEAD baseline test 는 scripts/baseline-test.sh 로만 — git stash/reset --hard/checkout HEAD -- 금지
  rationale: 2026-05-16 incident — baseline 검증용 git stash 가 reflog reset + working-tree 손상 유발
  consequence: ticket/commit diff 에 해당 명령이 baseline 목적으로 등장 → reviewer BLOCKER
놓치기 쉬움

D-XX는 단순 메모가 아니라 검토자에게 자동 BLOCKER 권한을 주는 레코드입니다. 새 "절대 규칙"을 만들고 싶으면 그냥 CLAUDE.md에 문장 추가가 아니라, ① grep으로 검출 가능 ② 위반 시 무조건 ship 금지 ③ 트레이드오프 종결 ④ OS 자체 영역 4조건을 통과시켜 D-XX로 등재해야 강제력이 생깁니다.

§4세션 경계의 기억 — 어떻게 까먹지 않나

AI 세션은 컨텍스트가 날아갑니다. dev-os는 세션 시작/종료 프로토콜로 기억을 파일에 고정합니다.

세션 시작 — Boot read map (한 번만 읽기)

session-start.md가 정한 부팅 시퀀스. 각 파일을 세션당 1회만 읽고 머릿속에 요약을 유지합니다(같은 파일을 offset 바꿔 다시 읽으면 "캐싱 실패" 신호).

  1. devos/AI.md (헌법)
  2. PROJECT_STATE.md
  3. CONTEXT.md
  4. QUEUE.yaml 전체 → id→status/owner 색인
  5. questions/QUEUE.md
  6. API_CONTRACT.md / UI_CONTRACT.md
  7. logs/ 최신 1–2개 (교차 컨텍스트)

에이전트별로 무엇을 읽는지는 AI.md의 "Session-start read map" 표가 정합니다. sub-agent는 슬림본 AI-core.md로 부팅하고(토큰 절약), 전문 AI.md는 필요할 때만 부분 읽기.

세션 종료 — 8축 자가질문 회고

session-end.md가 강제하는 종료 의식. 단순 "오늘 한 일" 요약이 아니라, 빠뜨린 게 없는지 8개 카테고리를 의무적으로 자문합니다(없으면 "없음"이라도 명시). 목적: 회고 누락률 통제.

#자가질문 카테고리
1user-AI 협업 마찰 (같은 지시 반복? 의도 오해?)
2DB/data 위험 (schema drift, 마이그레이션 누락, 데이터 손실)
3dev 환경 (token TTL, redirect loop, port 충돌, env 누락)
4잔재 파일 / .gitignore (임시파일, sentinel, working-tree 잔재)
5Wave 분기 / PM 결정 deferred (미해결 결정, 의존 unblock 누락)
6memory system — 이번에 발견한 반복 패턴이 memory 자격인가?
7vendor 정보 검증 — 정보를 직접 --help로 검증했나?

회고 헤더엔 revision_delta(v1 항목 수 → 최종 항목 수 변화율)를 기록합니다. 게이트가 아니라 메트릭 — "처음에 얼마나 빠뜨렸나"를 스스로 보게 만드는 압력 장치입니다.

장기 기억(MEMORY.md) write 트리거

세션을 넘겨 기억할 가치가 있는 것만 골라 적습니다(자동 강제 write 아님 — trigger만 명시):

  • 사용자가 기본 동작을 교정/override → 교정된 규칙 + 이유
  • 사용자가 선호·습관·품질 기준을 말함
  • 프로젝트 고유 관례 발견 (티켓 포맷, 훅 동작, 네이밍)
  • 라이브러리/API quirk를 조사로 확인
  • 회고에서 반복 패턴(≥2회) 발견

반대로 안 적는 것: 코드/git log/SSOT에서 쉽게 재발견되는 것, 일회성 세션 상태, 기존 memory의 중복(→ append 말고 update).

놓치기 쉬움

"기억"은 두 갈래입니다. (a) 프로젝트 상태는 devos/ 파일(공유, git 추적)에, (b) 사용자/협업 선호는 harness의 MEMORY.md(개인, git 밖)에 갑니다. 같은 사실을 양쪽에 중복 저장하지 않는 게 규칙입니다. learnings .jsonl은 또 다른 갈래로 "재사용 가능한 인사이트"의 append-only 로그입니다.

§5티켓 해부 — 일의 최소 단위

모든 구현은 QUEUE.yaml 안의 티켓입니다. 1 티켓 = 1 PR. 자기완결적이어야 합니다(구현 AI가 티켓+SSOT만 읽고 실행 가능).

- id: T-AUTH-LOGIN-01
  status: todo            # 새 티켓은 반드시 todo (디스패처는 이것만 집음)
  owner: CODEX            # BUILDER | CODEX | CLAUDE1(문서 한정)
  impl_owner: CODEX       # 구현 담당 (set 시 디스패처 타깃)
  test_owner: CODEX       # 테스트 담당 — cross-test 시 impl과 분리
  tdd: required           # required | skip | self-evident
  security_audit: true    # auth/결제/권한/외부입력 → 자동 강제 true
  cross_model: false      # 임계경로 → CODEX 2차 의견
  ethos: high             # 위험 키워드 감지 시 자동 → critical 분류
  skills_hint: [test-driven-development]   # 구현가에게 권장 skill
  goal: |  ...            # 무엇을 (행위 요구) — CLAUDE1 작성
  context: |  ...         # 왜 + 조사결과 (라이브러리 API 등) ← CLAUDE1의 리서치가 여기 park됨
  constraints: |  ...     # 기술 제약
  dod:                    # 완료 기준 — 각 항목 입력+기대출력
    - 'valid 자격증명 → 200 + access/refresh JWT'   # 성공
    - 'wrong password → 401 + "invalid credentials"' # ★에러 필수★
  files: [apps/api/src/auth/login.py, tests/test_auth_login.py]  # 소유권 범위
  verify: |  python3 -m pytest tests/test_auth_login.py -v
  deps: []               # 선행 티켓 (모두 done이어야 디스패치)
  gates: [pr-check]
  screenshot_tool: playwright   # (선택) user-outcome-review 게이트 활성

필드별 규칙 요약

필드규칙
dod성공 DOD가 있으면 에러 DOD 필수. "잘 작동함" 같은 모호 표현 금지. truthy 단언(assert response) 거부
files이 목록 밖의 파일을 건드리면 scope guard가 BLOCKER. NEW: 접두는 신규 파일 마커
tddrequired면 첫 커밋에 테스트 파일이 있어야(pr-check가 검사). self-evident는 waiver를 로그에 기록
owner 라우팅모호/감성 UI·새 UX → BUILDER. 백엔드/데이터/인프라/테스트/마이그레이션/정책 → CODEX. 문서·정책(devos/.claude) → CLAUDE1 직접
자동 강제security_audit·ethos:high는 키워드(삭제/결제/인증/권한/delete/payment/auth…) 감지로 자동 켜짐
금지어 (scope-reduction)

goal/dod/context에 v1으로 일단·static for now·TODO·임시·나중에·simplified·minimal·quick fix·hardcoded for now 등이 들어가면 티켓 진입 불가. 승인 직전 grep(정규식 1개)으로 자동 검사하고, PR review에서도 BLOCKER. 예외는 단 3가지(D-XX 분할 / spike·research / 외부제약)뿐.

모드 시대의 확장 필드 (TICKET_SCHEMA.md)

§6상태 머신 + 감사 추적

티켓은 정해진 상태값만 가집니다. 디스패처는 todo만 집습니다 — ready·pending·queued는 조용히 무시되니 쓰면 안 됩니다.

  - {status: blocked, reason: 'ticket_preflight_failed: missing file path: ...', actor: dispatcher, ts: ...}

actor가 dispatcher인 전이는 사람/AI가 아니라 엔진이 자동으로 막은 것입니다(예: preflight 실패). override: true는 정상 전이 규칙을 의도적으로 우회했다는 표식이고요. 티켓이 왜 이 상태인지 의문이면 _transition_history를 읽으면 전말이 나옵니다 — 채팅 로그를 뒤질 필요가 없습니다.

§10동작 모드 (Operating Modes) — 같은 일이라도 기준이 다르다

하나의 아이디어는 신뢰도가 오르며 Exploration → Productization → Production으로 이동합니다. 모드는 단순 라벨이 아니라 게이트가 막을지 경고만 할지를 바꾸는 실제 스위치입니다 — 티켓의 mode: 필드로 지정하고, server/dispatcher.py의 _gate_is_blocking()이 집행합니다. (출처: docs/OPERATING_DOCTRINE.md §4, docs/policy/MODE_GATE_MATRIX.md)

모드목적"완료"의 의미품질 자세
exploration빠르게 배우기 — 프로토타입·UX 스케치·기술 가능성 확인PM이 다음 제품 결정을 할 수 있다완성도보다 반복 속도. 테스트·보안 최소화 가능, visual review는 비차단
productization유망한 것을 production 준비 상태로 — 요구·리스크·acceptance 정리이 작업은 Production으로 구현할 준비가 됐다구현보다 기획 산출물 우선. user story·DOD·owner·files·gates 정의
production실제 제품에 남고 유지될 변경 — 사용자 노출·실데이터·인증/권한required 게이트·리뷰·웨이버·기록이 모두 닫힘좋은 개발팀처럼. 성공+실패 경로 테스트, 보안, 모든 state, migration/rollback
왜 모드가 핵심인가

"PM 친화적 = 낮은 품질"이 아닙니다. PM이 기술 세부를 일일이 검증하지 않으려면 시스템이 오히려 더 엄격해야 합니다 — 그 엄격함의 양을 모드가 조절합니다. exploration에서 빠르게 놀다가, production으로 승격되는 순간 자동으로 빡세집니다. 같은 코드라도 모드가 다르면 게이트 결과가 다릅니다.

Production 전용 규칙 — done은 구현자가 못 준다

mode: production 티켓엔 TICKET_SCHEMA.md가 강제하는 추가 규칙이 있습니다:

  • user_outcome 필수 (이 변경이 가능케 하는 사용자 결과)
  • DOD에 성공+실패 동작 필수 / files 명시 / deps는 QUEUE+ARCHIVE 횡단 해소
  • work_type: ui면 requires_visual_review: true (또는 inline 웨이버 레코드)
  • auth/payment/privacy/권한/destructive/외부입력 → security review 필수
  • 구현자는 code_ready까지만. done은 dispatcher가 required 게이트·독립 리뷰·보안/시각 리뷰·PM 결정·웨이버·기록을 확인한 뒤에만 부여 — "구현 끝 = 완료"가 아님 (§6)
놓치기 쉬움

지금 QUEUE의 활성 티켓 대부분은 mode 필드가 없습니다(레거시 = 전부 fail-closed로 동작). 반면 ARCHIVE의 실제 티켓 15건은 mode:를 씁니다. 즉 모드는 설계·구현되었지만 아직 모든 티켓에 일관 적용되지는 않은 상태입니다. 새 티켓을 만들 때 mode를 명시하면 그 티켓의 게이트 자세를 의도대로 제어할 수 있습니다.

Report-only가 실제로 강제되는 법 (fail-closed 모델)

dispatcher.py의 _gate_is_blocking가 구현한 정확한 규칙 — 오해 주의:

  • report-only = skip이 아니라 run-and-report. 게이트는 그대로 실행되고, 실패하면 세션 로그에 [REPORTED] <gate>(+ agent-review면 실제 WARNING verdict)를 남기고 done까지 통과시킨다. 안 돌리는 게 아니다.
  • 강등 조건은 둘 다 참일 때만: ① mode가 exploration/productization 그리고 ② 그 게이트가 명시적으로 soft 품질 게이트(tests/lint/reviewer/visual 등 정확 멤버십). 그 외 모든 게이트는 모든 모드에서 차단.
  • 기본값은 block: pr-check, 이름/명령에 secret·gitleaks·detect-secrets·trufflehog 등 hard 토큰이 든 게이트, 인식 못한/미래/리네임된 게이트는 전부 차단.
  • mode가 production/없음/unknown → 모든 게이트 차단 (레거시 무-mode 티켓은 원래의 all-blocking 유지).
  • production review FAIL은 차단하고 auto-WARNING으로 세탁 불가 (reject를 done으로 둔갑 금지).
항상 차단되는 안전 바닥 (exploration에서도)

모드와 무관하게 fail-closed: secret 노출 · destructive action · 개인정보/권한 사고 · file scope 위반 · PM 승인 없는 비가역 변경. pr-check는 secret 스캔을 내장하므로 어떤 soft allowlist에도 없음 — 모든 모드에서 막힙니다.

① Singleton PID 락 — 같은 프로젝트에 디스패처 둘 금지

dispatch 류 명령은 진입 시 원자적 파일 생성으로 락을 잡습니다. 두 번째 프로세스는 즉시 거부됩니다.

경로: ~/.os3/dispatcher.{project}.pid   # home 쓰기 불가 시 devos/.dispatcher.pid
획득: os.open(path, O_CREAT | O_EXCL | O_WRONLY, 0o600)   # EXCL = 이미 있으면 실패
내용: pid={현재 PID}\nstarted={ISO 타임스탬프}
해제: __exit__ + atexit + SIGINT/SIGTERM 핸들러 (3중 — 비정상 종료에도 정리)
충돌: FileExistsError → DispatcherSingletonError
  └ 기존 PID가 살아있나? os.kill(pid, 0) 로 확인
      ├ 살아있음 → "another instance is running (PID …)"  → 거부
      └ 죽었음   → "stale dispatcher lock found …"        → 안내(수동 정리)

fcntl이 없는 플랫폼이면 락을 건너뛰고 경고만 합니다(단일 사용자 환경 가정). stale 락은 자동 삭제하지 않고 사용자에게 알립니다 — 진짜 실행 중인 걸 죽이지 않기 위해.

② Dispatch snapshot — diff 기준선은 HEAD SHA 하나

구현 시작 직전, _capture_dispatch_snapshot()이 git rev-parse HEAD 한 줄로 현재 SHA를 박아둡니다. 이게 이 디스패치의 diff 기준선입니다 — 게이트(scope guard, contract sync, TDD first-commit)가 "이 SHA 이후 무엇이 바뀌었나"를 봅니다.

왜 stash가 아니라 SHA인가

baseline을 만들려고 git stash/worktree를 쓰지 않습니다. 그냥 현재 HEAD의 SHA 문자열을 기억할 뿐 — working tree를 전혀 건드리지 않습니다. 캡처 실패(예: git 오류)면 dispatch_snapshot_sha is None → 디스패치를 아예 시작하지 않습니다(기준선 없이 게이트를 돌리는 위험 차단). 이게 §13 사고 이후 자리잡은 read-only 자세의 코드 레벨 구현입니다.

③ 게이트 실패 → 1회 리트라이 (post-fail hook)

게이트가 실패하면 즉시 blocked가 아니라, 조건부 1회 자동 복구를 시도합니다:

gates 실패 → _attempt_retry 1. stderr 패턴 매칭으로 post-fail hook 탐색 (_find_post_fail_hook) └ 매칭 hook 없음 → 그대로 실패 반환 2. hook 액션 실행 (shell, <project-root> 치환, 기본 timeout 600s) └ hook 자체 실패 → stderr에 사유 누적 후 실패 반환 3. hook.retry == true 일 때만 → 서브프로세스 1회 재실행 4. 재실행 후 게이트 재평가 ├ 통과 → done ("passed after retry") └ 여전히 실패 → blocked ("gates still failing after retry")

리트라이는 무한 루프가 아니라 정확히 1회입니다. hook은 "알려진 일시적 실패"(예: 환경 셋업 누락)를 자동 교정하는 용도. retry: false hook은 교정만 하고 재실행은 안 합니다.

④ 에이전트 self-block · 쿼터 감지

  • self-block: 구현 에이전트가 핸드오프에서 Block: Q-XXX를 반환하면 _mark_agent_self_blocked → 즉시 blocked + agent_self_blocked: 사유(게이트까지 안 가고 short-circuit, 운영비 절감).
  • 쿼터 감지: CODEX subprocess stderr에서 쿼터 소진 패턴(try again at HH:MM AM/PM)을 정규식으로 잡아 quota_exhausted: {reset}로 기록 → 헛된 재시도 방지.

§17SSOT 동시성 — 파일 락 · 원자적 쓰기

QUEUE/ARCHIVE는 평범한 YAML 파일이라 동시 쓰기 = 손상 위험이 있습니다. server/ssot.py(67KB)가 advisory 파일 락 + 원자적 교체로 막습니다.

파일 락 (acquire_file_lock)

락 파일: .{name}.lock   (예: .QUEUE.yaml.lock)
방식: fcntl.flock(fd, LOCK_EX | LOCK_NB)   # 배타 + 논블로킹
재시도: BlockingIOError 시 0.1s 간격 폴링, 매 시도 stderr 로그
타임아웃: 30s 기본 (OS3_FILE_LOCK_TIMEOUT 로 조정) → 초과 시 LockTimeoutError
탈출구: fcntl 없는 플랫폼 → OS3_ALLOW_NO_LOCK=1 명시해야 진행(경고: race 가능)

원자적 쓰기 — 부분 읽기 방지

쓰기는 항상 임시 파일에 먼저 쓰고 os.replace()로 교체합니다(POSIX 원자적 rename). 그래서 다른 프로세스가 쓰는 도중의 반쪽짜리 파일을 읽는 일이 없습니다.

세 개의 분리된 락 — 디스패치와 archive의 공존

락 파일보호 대상핵심
.QUEUE.yaml.lockQUEUE.yaml 쓰기티켓 상태 전이 시 배타
.archive.lock디스패처 sentinelarchive 이관이 진행 중임을 표시. 디스패처는 ensure_archive_not_locked로 확인 — archive 도중 dispatch 금지

락 파일이 두 종류로 나뉜 건 의도된 설계입니다 — .archive.lock(이관 중이니 dispatch 멈춰)와 .archive-index.lock(인덱스만 다시 쓰는 중, dispatch는 계속 OK)를 분리해 불필요한 차단을 줄입니다. os3 lookup이 archive된 티켓도 찾는 건 ARCHIVE-INDEX 덕분이고, 그 인덱스도 원자적으로 재빌드됩니다.

5%는 일부러 빡세게 본다

가벼운 경로도 5% 확률로 critical 승격. 가끔 "왜 이 사소한 티켓에 security가 붙지?" 싶으면 audit_sample일 수 있음 — 정상.

b'는 평소 안 돌고, 돌면 이유가 있다

교차모델은 BLOCKER≥1 / WARNING≥3 / confidence<0.7 / 위험티켓일 때만. fallback rate 10–25%가 건강 구간.

검토자는 git을 망가뜨릴 수 없게 막혀 있다

2026-05-16 사고 후 reviewer는 stash/reset/checkout/clean/rebase가 금지. baseline은 scripts/baseline-test.sh(임시 worktree)로만.

금지어는 grep으로 걸린다

"임시", "나중에", "v1으로 일단"을 티켓에 적으면 승인 직전 자동 grep + review BLOCKER. 단축본을 못 만들게 막는 장치.

컨텍스트 25%면 스스로 멈추라는 신호를 받는다

context-monitor가 에이전트에게 CRITICAL을 주입 → 새 복잡 작업 시작 금지 + handoff 준비 권고. 길어지면 /clear + 4줄 handoff.

done은 쌓이면 자동으로 치워진다

세션 끝에 done≥1이면 os3 archive가 자동 호출되어 ARCHIVE로 이관(토큰 보호). archive돼도 lookup·verify로 계속 참조 가능.

같은 코드라도 mode가 게이트 결과를 바꾼다

mode: exploration이면 테스트·리뷰 실패가 [REPORTED] 경고로 내려가 done까지 통과. production이면 전부 차단. 티켓에 mode를 안 적으면 레거시=전부 차단. (§10–11)

production은 "구현 끝"이 "완료"가 아니다
