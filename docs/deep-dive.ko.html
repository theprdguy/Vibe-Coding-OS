0한 문단

Vibe Coding OS는 "AI에게 코드를 시키는 일"을 감사 가능한 상태 기계로 바꿉니다. 모든 진실은 devos/ 아래 파일에 있고, 일은 티켓으로만 움직이며, 누가 무엇을 왜 했는지는 티켓별 전이 이력과 세션 로그에 남습니다.

사용자(PM)가 의도를 말한다 → CLAUDE1이 티켓으로 분해하고 빠진 에러·엣지·보안 케이스를 캐묻는다 → 사용자가 plan을 승인한다 → 티켓 owner(builder 세션 내 / CODEX 외부)가 구현한다 → 베이스라인 게이트 + 읽기 전용 리뷰 보드가 검토한다 → 위험·불확실하면 다른 벤더(CODEX b')가 2차 의견 → BLOCKER 없으면 done, 있으면 blocked + 결정 큐.

13층 구조

세 겹입니다. 위 두 겹은 "무엇을 한다", 맨 아래가 "그걸 못 어기게 한다". 안전성의 대부분이 3층에서 옵니다.

층구성역할
1 · 진실파일 기반 SSOT (devos/)프로젝트가 무엇이고 어디까지 왔는지 기억
2 · 행위자계획가 + 구현가 2 + 읽기전용 리뷰 보드계획·구현·검토
3 · 강제도구 권한 · 훅 · 게이트 · grep · 잠긴 결정규칙을 "신뢰"가 아니라 "구조"로 막음

핵심 원리: 역할·권한이 분리되고, 권한은 도구 차원에서 제한된다. 검토자는 Edit/Write 툴이 없어 "고치고 싶어도 못 고친다" — 자기검토 객관성 붕괴가 원천 차단.

2에이전트

각자 다른 모델·다른 도구를 씁니다. 권한 분리는 선의가 아니라 구조로 강제됩니다.

CLAUDE1 (main)Opus계획·지휘

사용자와 대화. PRD를 티켓으로 분해·조사·라우팅. 코드를 직접 안 쓰고, 결과를 직접 OK 하지 않음 — 리뷰 보드로 넘김.

builderSonnetRead·Edit·Write·Bash

세션 내 구현가. 모호·감성 중요한 UI, 새 UX 흐름에 우선. 티켓 files 범위만 수정.

CODEX외부 CLIsubprocess·sandbox

플랫폼 구현가 — 백엔드/API/데이터/인프라/테스트/마이그레이션 — + 교차모델 2차 의견(b'). 다른 벤더라 Claude의 공통 맹점을 잡음.

reviewer · designer · securityREAD-ONLY

리뷰 보드(Opus/Sonnet). reviewer=적대적 PR 검토(DOD↔테스트·단언·범위·계약). designer=UI/UX(일관성·상태·접근성). security=OWASP+STRIDE, auth/결제/권한/외부입력 자동 발동. 셋 다 Edit/Write 툴 없음.

읽기전용이 핵심인 이유

대부분 팀은 "자기 PR 자기 머지 금지"를 정책으로 둡니다. 여기는 권한입니다 — 검토자에게 write 툴이 없어 발견만 하고 조용히 못 고칩니다. 구조적 객관성.

3티켓 — 일의 단위

모든 구현은 devos/tasks/QUEUE.yaml의 티켓입니다. 1 티켓 = 1 PR. 자기완결적 — 구현가가 티켓 + SSOT만 읽고 실행 가능해야 합니다.

- id: T-AUTH-LOGIN-01
  status: todo            # 디스패처는 todo만 집음
  owner: CODEX            # BUILDER | CODEX
  mode: production        # 게이트 자세 (§6)
  tdd: required
  security_audit: true    # auth/결제/권한 자동 강제
  goal: |   POST /auth/login
  dod:
    - 'valid 자격증명 -> 200 + access/refresh JWT'     # 성공
    - 'wrong password -> 401 + "invalid credentials"'  # 에러(필수)
  files: [apps/api/src/auth/login.py, tests/test_login.py]
  verify: |  python3 -m pytest tests/test_login.py -v
  gates: [pr-check]

DOD 규칙: 성공 DOD가 있으면 에러 DOD 필수. 비개발자가 happy path만 적는 빈틈을 시스템이 메우는 장치.

상태 머신 + 감사 추적

상태: todo doing code_ready needs_pm done blocked parked. 전이마다 actor·reason·timestamp가 _transition_history에 누적 — 티켓이 왜 이 상태인지 채팅이 아니라 파일에서 재구성.

4흐름

사용자 --PRD--> CLAUDE1
               | intake 체크리스트 (빠진 에러/엣지/보안 강제 질문)
               | user journey 추출 -> 티켓 분해
               v
       plans/pending/  --사용자: os3 approve-->  tasks/QUEUE.yaml
                                                   |
                              os3 dispatch T-XXX   v
          owner=BUILDER -> 세션 내 sub-agent      owner=CODEX -> subprocess
                                 | 구현
                                 v
          os3 pr-check (시크릿 · 계약 · 범위 · 세션로그 · TDD)
                                 v
          리뷰 보드 (reviewer/security/designer — 읽기전용, 티켓 유형별)
                                 v
          불확실/위험?  -> b' 교차모델 (CODEX 2차 의견)
                                 v
       BLOCKER -> blocked + 질문 큐 (사용자 결정)
       그 외   -> done -> archived (감사 추적 보존)

모든 디스패치는 표준 결과 헤더(어느 경로·어느 모델이 만들고·누가 검토·verdict)를 출력 — 매번 묻지 않게.

5디스패치 (결정적)

단계동작
1 · lookup+preflightstatus==todo? deps 모두 done? files 존재?
2 · owner 라우팅BUILDER → 세션 내 / CODEX → subprocess
3 · 구현구현가가 자기 컨텍스트에서 작업
4 · pr-check시크릿 · 계약 동기화 · 범위 가드 · 세션 로그 · TDD 첫커밋
5 · 리뷰 체인티켓 분류 → 필요한 읽기전용 검토자 spawn (+5% 무작위 표본은 가벼운 경로도 승격)
6 · fan-in + b'verdict 취합; 정량 조건 충족 시 CODEX 2차 의견
7 · 최종BLOCKER 우선 판정 → 상태 + 로그 + 표준 헤더
결정적 엔진 + 규율적 래퍼

엔진(dispatcher.py)은 코드로 결정적 — singleton 락, HEAD-SHA diff 기준선(git stash 안 씀), 게이트, 1회 리트라이. 그 위 오케스트레이션(분류·b' 트리거)은 CLAUDE1이 위 프로토콜대로 실행.

6동작 모드

하나의 아이디어가 신뢰도가 오르며 exploration → productization → production으로 이동. 모드가 게이트 자세를 정하고 디스패처가 강제합니다.

게이트explorationproductizationproduction
Tests선택/report-only전략 필수필수
Reviewerreport-only권장필수 · reject 차단
Security시크릿/auth만리스크 식별트리거 시 필수
성공+실패 DOD선택필수필수

Report-only는 fail-closed: 모드가 exploration/productization 이고 명시적 soft 게이트일 때만 완화. 시크릿·범위 위반·destructive·미인식 게이트는 모든 모드에서 차단. Production 예외는 기록된 웨이버(승인자·위험·follow-up) 필요.

7안전 — 신뢰가 아니라 구조

🔒 권한으로 강제된 객관성

검토자는 읽기전용 도구. 계획가는 훅이 apps/ packages/ scripts/ tests/ 쓰기를 차단. 둘 다 약속이 아니라 도구 차원의 사실.

⚖️ Iron Laws (전 에이전트)

실패 테스트 없이 prod 코드 없다 · 근본원인 없이 수정 없다 · 새 증거 없이 "완료" 없다 · 적대적 검토 없이 머지 없다 · 범위축소 어휘 없다(grep 강제).

🚪 항상 켜진 안전 바닥

시크릿 노출·owner mismatch·범위 위반·미해결 deps·dirty-tree destructive는 모드 무관 차단, 일반 웨이버로 우회 불가.

🤝 교차모델 안전망(b')

검토자 불확실·BLOCKER·고위험 티켓이면 다른 벤더가 2차 의견. 평소엔 비용 0.

사고 → 잠긴 결정

무언가 부서지면 그 수정은 강제 규칙이 됩니다 — 위반이 자동 reviewer BLOCKER인 D-XX Locked Decision. 그래서 재발 못 함. 규칙집은 위시리스트가 아니라 과거 사고의 기록.

8LLM이 OS를 읽는 법 (SSOT)

모든 상태가 파일. 세션 시작 시 다시 읽으므로 공유 채팅창 드리프트가 없습니다. 충돌 시 위가 강함:

1. PROJECT_STATE.md        마일스톤·현재 동작
2. API_CONTRACT / UI_CONTRACT   인터페이스 계약 (코드보다 우선)
3. ADR/*                   아키텍처 결정
4. tasks/QUEUE.yaml        활성 티켓
5. 코드
6. 세션 로그
7. 채팅 로그               (가장 안 믿음)

Contract-first: API/UI 동작 변경 시 계약 문서를 먼저 고침; 불일치는 reviewer가 BLOCKER.

9host-OS 모델

엔진·에이전트·도리는 호스트 레포에 한 번만 존재. 각 제품은 projects/ 아래 독립 git 레포로, 자기 코드와 task 상태만 보유.

~/dev-os/            호스트 = 엔진 · 도리 · 공유 .claude (git 1개)
|-- server/  bin/os3  .claude/  devos/
`-- projects/        (호스트가 git-ignore — 각자 독립 레포)
    |-- product-a/   apps/ packages/ + 자기 devos/tasks · PROJECT_STATE · .os3.yaml
    `-- product-b/   ...
  • 엔진은 1곳 — 고치면 전 프로젝트 적용. sync 없음.
  • 프로젝트는 독립 — 자기 레포·상태, 따로 배포.
  • 진입은 하나 — os3 open <name>이 호스트 설정을 달고 진입.

단일 프로젝트로 써도 됨 — devos/를 상태로 쓰고 projects/ 무시. host-OS는 여러 제품으로 확장하는 방식.

10OS-feedback 루프

OS는 자기를 demand-pulled로 개선합니다: 제품 작업 중 마찰을 os3 feedback "..."로 devos/os-feedback/INBOX.md에 포착하고, 세션 시작 시 검토하고, 주기적 consolidation 패스에서 티켓으로 전환.

왜 load-bearing인가

OS 개선이 "제품 마찰에서" 온다면 그 마찰을 포착해야 합니다 — 안 그러면 채팅에서 한 번 말하고 증발. 반응적 수정은 부딪힌 것만 잡고, 주기적 consolidation은 안 밟는 잠재 부채를 잡습니다.

11명령어

os3 open <project>        프로젝트 세션 진입 (호스트 설정 주입)
os3 status / queue       상태 · 활성 티켓
os3 approve / reject     플랜 승인·반려
os3 dispatch T-XXX       티켓을 owner로 라우팅
os3 pr-check             베이스라인 게이트
os3 archive              done 티켓 → ARCHIVE.yaml
os3 dashboard            로컬 읽기전용 칸반 (127.0.0.1:8787)
os3 feedback "..."       OS 마찰을 INBOX에 포착
